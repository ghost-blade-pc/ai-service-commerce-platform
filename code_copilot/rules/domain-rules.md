# 领域规则

## 工作区领域边界

当前工作区至少包含两个业务上下文：

- `group-buy-market`：拼团营销、活动、折扣计算、锁单、结算、退款相关营销处理。
- `s-pay-mall-ddd`：商城订单、支付、退款、登录、与拼团营销服务交互。

不要假设两个上下文共享同一个数据库、同一个领域模型或同一个事务边界。跨项目协作必须通过明确的 API、MQ、Gateway、Port 或任务补偿表达。

## 跨项目变更要求

跨 `group-buy-market` 与 `s-pay-mall-ddd` 的需求，Spec 必须写清：

- 调用方向：谁调用谁，入站还是出站。
- 同步/异步方式：REST、MQ、Job、人工补偿或其他。
- 契约：path、method、DTO、Response、Exchange、Routing Key、Queue、Message。
- 幂等：幂等键、重复请求、重复消息、乱序消息处理方式。
- 状态：订单、支付、退款、拼团锁单、拼团结算的前置状态和目标状态。
- 失败处理：超时、重试、降级、补偿、告警和日志。

## 高风险业务

以下变更默认高风险，进入 `/apply` 前必须在 `spec.md` 中明确验收条件和人工确认点：

- 支付成功回调、退款、资金金额、优惠金额、结算金额。
- 订单创建、锁单、支付、关闭、退款、状态补偿。
- 拼团活动、拼团锁单、成团结算、退单后营销状态回滚。
- RabbitMQ 消息生产/消费、重复消费、死信、重试。
- MyBatis SQL、表字段、索引、初始化脚本。
- 对外 REST 契约、鉴权、签名、超时与重试。

## 禁止事项

- 禁止在未 Research 的情况下发明表、字段、队列、topic、接口路径或状态枚举。
- 禁止在 `domain` 中直接依赖 Controller、DAO、Mapper、HTTP 客户端、RabbitMQ 注解。
- 禁止把跨项目一致性问题伪装成单项目内部事务。
- 禁止在已完成 change 中继续塞入新需求；运行中发现的新问题应拆分新的 `changes/<change-id>`。
